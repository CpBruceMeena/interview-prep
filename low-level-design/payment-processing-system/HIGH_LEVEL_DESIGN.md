## 3. KEY COMPONENTS & INTERVIEW Q&A

### Payment Service (Python/Go)
- Validation pipeline: amount вҶ’ currency вҶ’ payment method вҶ’ fraud checks
- Gateway routing by currency/amount/region
- Idempotency вҖ” the most critical feature

**рҹ”ҙ Interview Question:** *"How do you guarantee idempotency in payment processing?"*

**вң… Answer:** Every payment request includes an `Idempotency-Key` header (UUID generated by the client). Flow:
```python
def process_payment(idempotency_key, amount, ...):
    # 1. Check if this key was already processed
    existing = redis.get(f"idemp:{idempotency_key}")
    if existing:
        return existing  # Return same result вҖ” never double-charge!
    
    # 2. Process with gateway
    result = gateway.charge(amount, ...)
    
    # 3. Store result with TTL (24 hours)
    redis.setex(f"idemp:{idempotency_key}", 86400, result)
    return result
```
The key insight: network retries after a successful charge should NOT reprocess. The idempotency key ensures the second `POST` returns the first `200 OK` вҖ” payment only processed once.

### Fraud Detection (Chain of Responsibility)
- Amount threshold check
- Velocity check (too many transactions in short time)
- Geolocation/IP anomaly detection
- ML model scoring (real-time, <50ms)

**рҹ”ҙ Interview Question:** *"How do you balance fraud prevention with minimizing false positives?"*

**вң… Answer:** Layered approach with configurable thresholds:
1. **Rules engine:** Hard blocks (amount > $10K, >10 transactions/minute). Low false positives.
2. **ML model:** Trained on historical chargebacks. Scores 0-100. Threshold tuned for 95% recall.
3. **Manual review:** Transactions scoring 80-95 flagged for manual review (1 hour SLA).
4. **A/B testing:** New rules deployed to 10% of traffic first, monitor false positive rate.
5. **Feedback loop:** Confirmed fraud вҶ’ retrain model. False positive вҶ’ adjust threshold.

Target: Catch 95% of fraud with <1% false positive rate.

### Settlement Engine
- Daily batch settlement calculation
- Payout to merchants (T+1, T+3, weekly)
- Reconciliation against bank statements

**рҹ”ҙ Interview Question:** *"How does daily merchant settlement work?"*

**вң… Answer:**
```python
def daily_settlement(merchant_id, date):
    # Calculate
    gross = sum(charges for merchant on date)
    fees = sum(gross * merchant.fee_percentage)
    refunds = sum(refunds processed on date)
    net = gross - fees - refunds
    
    # Generate ACH/NEFT file
    bank_file = generate_ach(merchant.bank_account, net)
    
    # Upload to bank
    bank_connector.upload(bank_file)
    
    # Record for reconciliation
    Settlement.create(merchant_id, date, gross, fees, net)
```

## 4. DATA MODEL

```sql
CREATE TABLE payments (
    id UUID, idempotency_key TEXT UNIQUE,
    amount DECIMAL(12,2), currency TEXT, status TEXT,
    gateway_tx_id TEXT, created_at TIMESTAMP
);
CREATE TABLE refunds (
    id UUID, payment_id UUID, amount DECIMAL(12,2),
    reason TEXT, status TEXT, created_at TIMESTAMP
);
CREATE TABLE settlements (
    id UUID, merchant_id UUID, period_start DATE,
    period_end DATE, gross DECIMAL(12,2), fees DECIMAL(12,2),
    net DECIMAL(12,2), status TEXT
);
CREATE TABLE fraud_checks (
    id UUID, payment_id UUID, check_type TEXT,
    passed BOOLEAN, reason TEXT, score DECIMAL(5,2)
);
```

## 5. TRADE-OFF ANALYSIS

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Idempotency | Client-generated key | Simplest, no coordination needed |
| Gateway routing | Geo-based (lowest cost) | Reduces cross-border fees by 30% |
| Settlement | T+1 batch | Balances cash flow with operational cost |
| Storage | SQL + Redis | ACID for transactions, fast reads for idempotency |
