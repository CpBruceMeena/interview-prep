Designing a Distributed Rate Limiter

A deep-dive into throttling strategies asked in almost every senior backend / system design interview loop — covering algorithms, storage trade-offs, and distributed edge cases.

Why Rate Limiting Matters

Rate limiting protects services from abuse, noisy-neighbor traffic, and cascading failures. Interviewers use this problem to evaluate how you reason about trade-offs under concurrency and partial failure.

  • Prevents a single client from exhausting shared resources.
  • Enables tiered pricing (free vs. pro API quotas).
  • Acts as a first line of defense against DDoS-style bursts.
“A good rate limiter is invisible to well-behaved clients and unforgiving to bad ones.”

💡 Interview Tip

Always default to the Token Bucket algorithm in interviews — it naturally supports burst traffic while enforcing a steady average rate, and it's simple to reason about on a whiteboard.

Token Bucket Implementation

Each user gets a bucket that refills at a fixed rate. A request is allowed only if a token is available.

Python 
class TokenBucket:
    def __init__(self, capacity, refill_rate):
        self.capacity = capacity
        self.tokens = capacity
        self.refill_rate = refill_rate  # tokens per second
        self.last_check = time.time()

    def allow_request(self):
        now = time.time()
        elapsed = now - self.last_check
        self.tokens = min(self.capacity, self.tokens + elapsed * self.refill_rate)
        self.last_check = now

        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False
⚠️ Common Trap

Forgetting to make the bucket thread-safe is the #1 mistake candidates make. In a distributed setup, back this with Redis + a Lua script for atomic check-and-decrement.

Distributing It Across Nodes

A single in-memory bucket doesn't scale across multiple API gateway instances. Two common patterns solve this:

  • Centralized store (Redis): Single source of truth, atomic ops via Lua scripts, slight network latency per request.
  • Sticky sessions + local buckets: Lower latency, but uneven distribution if load balancer isn't consistent-hash aware.
JavaScript 
// Redis Lua script executed atomically
const luaScript = `
  local tokens = tonumber(redis.call('get', KEYS[1]) or ARGV[1])
  if tokens > 0 then
    redis.call('decrby', KEYS[1], 1)
    return 1
  end
  return 0
`;

async function allowRequest(userId) {
  const result = await redis.eval(luaScript, 1, `bucket:${userId}`, capacity);
  return result === 1;
}
🔎 Deep Dive

For extremely high QPS systems (Cloudflare-scale), rate limiting is often approximated using probabilistic data structures like Count-Min Sketch to avoid per-key memory blow-up — a great follow-up point to raise if the interviewer pushes on scale.

Likely Follow-Up Questions

  • How do you rate-limit by IP and API key simultaneously?
  • What happens to the limiter during a Redis failover?
  • How would you expose remaining-quota headers to clients?
